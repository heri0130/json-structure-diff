# JSON 구조 분석기 (JSON Structure Analyzer)

> 폐쇄망/로컬용 개발 보조 도구로, 두 가지를 한다 (교육 실습 과제):
> - **비교 모드**: 두 JSON(예: API 응답 변경 전/후)의 **구조(스키마) 변경** — 키 추가·삭제, 값의 타입 변경 — 을 트리로 보여준다.
> - **명세서 모드**: JSON 하나의 필드 명세서(경로·타입·예시)를 자동 생성한다.
>
> 비교는 값 자체의 변경(예: `89000 → 79000`)을 **타입이 같으면 동일**로 본다. "스키마가 바뀌었나"에 집중한다.
> (저장소·패키지 이름은 초기 `json-structure-diff`를 유지)

이 문서는 PRD 인터뷰에서 시작해 구현 완료까지 반영한 결과물입니다. (1~5장은 PRD, 6장은 상세 설계)

상태: **PRD 완성 · MVP + 추가 기능 구현 완료** (TDD, 단위 테스트 37개 green)

**주요 기능**: 모드 전환(비교 / 명세서) · 두 JSON 구조 비교(키 추가·삭제·타입 변경) · 배열은 요소 스키마 기준(개수·순서·값 무시) · 변경 트리 시각화(값/타입 표시) · 변경된 것만 보기 · 단일 JSON 명세서 생성 · JSON 유효성 검사(JSONC 주석 허용) · 결과 클립보드 복사 · 모드별 예시 5종 · 테마 4종(라이트/다크/공주님/왕자님)

## 1. 프로젝트 개요 (왜 만드는가)

- **해결하려는 문제**: 폐쇄망이라 온라인 JSON 비교 도구를 못 쓰는데, 두 JSON(예: API 응답 변경 전/후)의 **구조 변경**(키 추가·삭제, 타입 변경)을 눈으로 일일이 대조하기 번거롭고 실수가 난다.
- **배경**: 실습 과제로 만들 겸, 실무에서 Swagger 같은 API 문서 도구가 없어 API 명세를 엑셀에 수기로 정리 중이라 응답 구조 변경 대조가 힘들다.
- **기대 변화**: (현재) 엑셀에 옮겨 적고 눈으로 비교, 바뀐 필드 놓침 → (앞으로) 두 JSON을 붙여넣으면 키 추가·삭제·타입 변경이 색으로 즉시 구분됨(값만 다른 노이즈는 자동 제외).

## 2. 사용자 분석 (누가 쓰는가)

- **주 사용자**: 나 자신 (우선 실습용). 확장 시 변경 내용을 프론트 개발자에게 공유.
- **사용 맥락**: 백엔드 API 스펙 변경 → 엑셀 명세 갱신 → 프론트에 공유하기 전, 무엇이 바뀌었는지 확인할 때.
- **현재 해결 방식과 한계**: 엑셀 이전 버전 눈대조 / 변경 이력 미기록 / Postman 응답 복사 후 수동 비교 — 세 방식 다 사용 중이라 놓치기 쉽고 시간이 오래 걸린다.

## 3. 기능 요구사항 (무엇을 하는가)

- **핵심 기능** (우선순위 순)
  1. **두 JSON 구조 비교** — 키 추가/삭제, 타입 변경(number↔string 등)을 찾아냄. 값만 다르고 타입이 같으면 동일로 본다. 배열은 요소 스키마 기준(개수·순서·값 무시)
  2. **결과 시각화 + 값/타입 표시** — 변경 종류를 색/트리로 구분, 변경 노드는 `before(타입) → after(타입)`로 표시(타입이 바뀌면 강조)
  3. **JSON 유효성 검사** — 깨진 JSON이면 어디가 틀렸는지 안내
- **추가 구현 완료**: 필드 명세서 자동 생성, 결과 마크다운 내보내기, 변경된 것만 보기
- **다음 단계(보류)**: 변경 이력 저장
- **차별점**: 폐쇄망 로컬 실행 + **구조(스키마) 변경에 집중**(값 노이즈 제외) + 엑셀 명세/공유 워크플로우 연계 (온라인 JSON diff 도구와의 차이)

## 6. 상세 설계 (확장)

### 6.1 사용자 여정 (User Journey)

1. 브라우저로 도구를 연다 (로컬 실행). 상단에서 **비교 / 명세서** 모드를 고른다.
2. 입력 — (비교) 예시 칩으로 샘플을 불러오거나 두 입력창에 JSON을 직접 붙여넣는다(왼쪽=변경 전, 오른쪽=변경 후). (명세서) JSON 하나를 붙여넣는다.
3. **분석하기**(명세서 모드는 **명세서 만들기**) 클릭 → 먼저 유효성 검사. 깨진 JSON이면 위치를 알려주고 멈춘다.
4. 결과 확인 — (비교) **변경 사항** 탭: 키 추가(초록)/삭제(빨강)/타입 변경(노랑)/동일(회색) 트리 + `before → after` 값/타입 + 요약("변경된 것만 보기"로 동일 항목 숨김 가능), **명세서** 탭: 변경 후 구조의 필드 경로·타입·예시. (명세서 모드) 명세서 표만. 결과는 마크다운으로 클립보드 복사 가능.
5. 결과를 보고 엑셀 명세를 갱신하거나 프론트 개발자에게 변경점을 전달한다.

### 6.2 기능별 입출력 사양

**① 두 JSON 구조 비교**
- 입력: 파싱된 두 JSON 값(old, new)
- 동작: 키 경로를 재귀로 순회. 같은 키는 **타입**으로 비교(타입 같으면 unchanged, 다르면 changed), 키 유무로 added/removed 판정. 배열은 요소들을 **대표 스키마로 병합**해 그 스키마끼리 비교(개수·순서·값 무시)
- 출력: 차이 노드 목록 `{ path, status, oldValue, newValue }` (배열 요소 스키마 경로는 `list[].필드`)

**② 결과 시각화 + 값/타입 표시**
- 입력: ①의 diff 결과
- 동작: 트리로 렌더링, 상태별 색상 부여, 변경 노드에 `before(타입) → after(타입)` 표기(타입 변경 시 강조). 하위 펼치기/접기, "변경된 것만 보기" 토글로 동일 노드 숨김
- 출력: 화면 트리 + 요약(추가 N / 삭제 N / 변경 N)

**③ JSON 유효성 검사**
- 입력: 입력창 원문 텍스트 2개
- 동작: `JSON.parse` 시도, 실패 시 에러 메시지와 가능한 위치(라인/오프셋) 안내. 빈 입력도 안내
- 출력: 유효/무효 + 사용자용 메시지 (앱은 죽지 않음)

### 6.3 개발 우선순위 (기획 당시 계획)

> 아래는 기획 시점의 계획이다. 실제로는 MVP 이후 **배열 비교(요소 스키마 기준), 결과 마크다운 내보내기, 명세서 모드**까지 완료했고, 비교 기준도 **값 → 구조(키/타입)**로 발전했다.

- **MVP**: ③유효성 검사 → ①비교(평면 + 중첩 객체) → ②기본 트리 표시 + 색상. 단위 테스트 green.
- **다음**: 배열 비교, 값/타입 표기 다듬기, 요약 카운트. → **완료**
- **추후**: 결과 내보내기 → **완료** / 변경 이력 저장 → **보류**

### 6.4 위험 요소 (대응 결과)

1. **중첩·배열 비교 복잡도** — 배열 순서/인덱스 처리가 까다로움 → 인덱스 비교를 거쳐 **요소 스키마 병합 비교**로 정리(순서·개수·값에 흔들리지 않음).
2. **대용량 JSON 성능** — 큰 응답은 트리 렌더링이 느려질 수 있음 → 동작 우선, 가상 스크롤 등은 추후 과제로 남김.
3. **마감 압박에 따른 스코프 욕심** — MVP를 먼저 고정하고 추가 기능은 단계적으로 커밋해 미완성 위험을 줄임.

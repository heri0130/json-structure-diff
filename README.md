# JSON 구조 비교기 (JSON Structure Diff)

> 두 JSON(예: API 응답 변경 전/후)을 붙여넣으면, 어떤 키가 추가·삭제·변경됐는지
> 트리로 보여주는 폐쇄망/로컬용 개발 보조 도구. (교육 실습 과제)

이 문서는 PRD 인터뷰를 진행하며 단계별로 계속 업데이트됩니다.

진행 상태: `1단계 ✅` · `2단계 ✅` · `3단계 ✅` · `4단계 ✅` · `5단계 ✅` — **PRD 완성** · **MVP + 추가 기능 구현 완료 (TDD, 테스트 24개 green)**

**주요 기능**: 두 JSON 비교(중첩·배열 인덱스 단위) · 변경 트리 시각화(값/타입 표시) · JSON 유효성 검사 · 필드 명세서 자동 생성 · 결과 탭 전환(변경 사항/명세서) · 예시 5종 · 테마 4종(라이트/다크/공주님/왕자님)

## 1. 프로젝트 개요 (왜 만드는가)

- **해결하려는 문제**: 폐쇄망이라 온라인 JSON 비교 도구를 못 쓰는데, 두 JSON(예: API 응답 변경 전/후)의 차이를 눈으로 일일이 대조하기 번거롭고 실수가 난다.
- **배경**: 실습 과제로 만들 겸, 실무에서 Swagger 같은 API 문서 도구가 없어 API 명세를 엑셀에 수기로 정리 중이라 응답 구조 변경 대조가 힘들다.
- **기대 변화**: (현재) 엑셀에 옮겨 적고 눈으로 비교, 바뀐 필드 놓침 → (앞으로) 두 JSON을 붙여넣으면 추가/삭제/변경이 색으로 즉시 구분됨.

---

## 2. 사용자 분석 (누가 쓰는가)

- **주 사용자**: 나 자신 (우선 실습용). 확장 시 변경 내용을 프론트 개발자에게 공유.
- **사용 맥락**: 백엔드 API 스펙 변경 → 엑셀 명세 갱신 → 프론트에 공유하기 전, 무엇이 바뀌었는지 확인할 때.
- **현재 해결 방식과 한계**: 엑셀 이전 버전 눈대조 / 변경 이력 미기록 / Postman 응답 복사 후 수동 비교 — 세 방식 다 사용 중이라 놓치기 쉽고 시간이 오래 걸린다.

## 3. 기능 요구사항 (무엇을 하는가)

- **핵심 기능** (우선순위 순)
  1. **두 JSON 비교** — 붙여넣은 두 JSON의 추가/삭제/값변경을 찾아냄
  2. **결과 시각화 + 값 표시** — 변경 종류를 색/트리로 구분, 키 옆에 before/after 실제 값을 함께 표시
  3. **JSON 유효성 검사** — 깨진 JSON이면 어디가 틀렸는지 안내
- **다음 단계(보류)**: 결과 내보내기(텍스트/마크다운), 변경 이력 저장
- **차별점**: 폐쇄망 로컬 실행 + 변경 키 옆 실제 값 표시 + 엑셀 명세/공유 워크플로우 연계 (온라인 JSON diff 도구와의 차이)

## 6. 상세 설계 (확장)

### 6.1 사용자 여정 (User Journey)

1. 브라우저로 도구를 연다 (로컬 실행).
2. (선택) 상단 예시 칩으로 샘플을 불러오거나, 두 입력창에 JSON을 직접 붙여넣는다 — 왼쪽=변경 전(old), 오른쪽=변경 후(new).
3. **분석하기** 클릭 → 먼저 유효성 검사. 깨진 JSON이면 어느 위치가 문제인지 알려주고 멈춘다. 둘 다 유효하면 변경 사항과 명세서를 함께 생성.
4. 결과 영역의 탭으로 확인 — **변경 사항**: 추가(초록)/삭제(빨강)/변경(노랑)/동일(회색) 트리 + `before → after` 값/타입 + 요약, **명세서**: 변경 후 구조의 필드 경로·타입·예시.
5. 결과를 보고 엑셀 명세를 갱신하거나 프론트 개발자에게 변경점을 전달한다.

### 6.2 기능별 입출력 사양

**① 두 JSON 비교**
- 입력: 파싱된 두 JSON 값(old, new)
- 동작: 키 경로를 재귀로 순회하며 중첩 객체/배열까지 비교. 경로별 상태(added / removed / changed / unchanged) 판정
- 출력: 차이 노드 목록 `{ path, status, oldValue, newValue }`

**② 결과 시각화 + 값 표시**
- 입력: ①의 diff 결과
- 동작: 트리로 렌더링, 상태별 색상 부여, 변경 노드에 `before → after` 표기. (동일 노드는 접기 옵션)
- 출력: 화면 트리 + 요약(추가 N / 삭제 N / 변경 N)

**③ JSON 유효성 검사**
- 입력: 입력창 원문 텍스트 2개
- 동작: `JSON.parse` 시도, 실패 시 에러 메시지와 가능한 위치(라인/오프셋) 안내. 빈 입력도 안내
- 출력: 유효/무효 + 사용자용 메시지 (앱은 죽지 않음)

### 6.3 개발 우선순위 (마감 16시 기준)

- **MVP (오늘 반드시)**: ③유효성 검사 → ①비교(평면 + 중첩 객체) → ②기본 트리 표시 + 색상. 단위 테스트(비교 로직) green.
- **다음 (시간 남으면)**: 배열 비교(인덱스 기준), `before → after` 값 표기 다듬기, 요약 카운트.
- **추후 (보류)**: 결과 내보내기(텍스트/마크다운), 변경 이력 로컬 저장.

### 6.4 위험 요소

1. **중첩·배열 비교 복잡도** — 특히 배열은 순서/인덱스 처리가 까다로워 시간을 많이 잡아먹을 수 있음 → MVP는 객체 중첩까지만, 배열은 "다음"으로.
2. **대용량 JSON 성능** — 큰 응답을 붙여넣으면 트리 렌더링이 느려질 수 있음 → 우선 동작 우선, 가상 스크롤 등은 추후.
3. **마감 압박에 따른 스코프 욕심** — 기능 욕심내다 미완성 위험 → MVP 우선 고정, 나머지는 명시적으로 보류.
